# 📷 آتلیه معین — سیستم مدیریت جامع

سیستم مدیریت حرفه‌ای آتلیه عکاسی و فیلمبرداری عروسی، ساخته‌شده با **Next.js 15** و **TypeScript**.

---

## ✨ امکانات

### مدیریت مراسمات
- رزرو سریع مراسم از طریق تقویم جلالی یا فرم مستقل
- پشتیبانی از چند مراسم در یک روز
- قرارداد کامل با جزئیات: نوع، تاریخ، ساعت، آدرس، مبلغ
- نمایش مبالغ با فرمت فارسی (جداکننده هزارتایی) و تبدیل عدد به حروف
- مدیریت وضعیت: رزرو / جاری / تکمیل / لغو

### مدیریت کارمندان
- پروفایل کامل با ساعت کاری، ساعت حضور، توضیحات شغلی
- لیست وظایف روزانه با نوار پیشرفت رنگی
- تخصیص کارمندان به هر مراسم با شرح نقش
- صندوق پیام ساده: ارسال پیام کارمند به مدیریت + پاسخ مدیر در پنل شخصی
- وظایف مراسم زمان‌دار: قبل از موعد، کاربر نمی‌تواند «انجام شد» ثبت کند

### تقویم جلالی هوشمند
- نشان‌دهی رنگی روزها: آبی = ۱ مراسم، بنفش = ۲+ مراسم
- نشان مناسبت‌های رسمی ایرانی (نقطه قرمز + tooltip)
- رزرو سریع با کلیک روی روزهای خالی (مدیر)

### احراز هویت و نقش‌ها
- ورود امن با JWT ذخیره‌شده در HttpOnly Cookie
- نقش‌ها: `admin` / `accountant` / `employee`
- ریدایرکت خودکار بر اساس نقش

### تجربه کاربری
- **Dark Mode** بدون flicker با `next-themes`
- طراحی RTL کاملاً ریسپانسیو (موبایل / تبلت / دسکتاپ)
- اعلان‌های toast با Sonner (پشتیبانی از dark mode)
- PWA Ready با manifest و آیکون‌های مناسب
- پروفایل شخصی `/profile` برای همه کاربران احراز هویت‌شده
- منوهای بدون دسترسی در داشبوردها نمایش داده نمی‌شوند (Hide instead of disabled)

## � قوانین دسترسی و RBAC

سیستم کنترل دسترسی مبتنی بر نقش (Role-Based Access Control) در این پروژه چند لایه دارد:

### انواع نقش

| نوع | شرح |
|-----|------|
| **سیستمی (`isSystem: true`)** | نقش‌های پایه‌ای که توسط سیستم ایجاد شده‌اند (`admin`، `accountant`، `employee`). اگر کاربری به آن‌ها اختصاص داشته باشد قابل حذف نیستند؛ اگر هیچ کاربری نداشته باشند، مدیر می‌تواند آن‌ها را حذف کند. |
| **سفارشی** | نقش‌هایی که مدیر ایجاد می‌کند و مجوزهای آن‌ها از طریق ماتریس مجوزها تنظیم می‌شود. |

نقش `admin` و `accountant` همیشه به همه‌چیز دسترسی دارند (مجوز `*`). سایر کاربران فقط به مجوزهایی که به نقش‌شان داده شده دسترسی دارند.

### گروه‌بندی مجوزها

| گروه | کلیدهای مجوز |
|------|---------------|
| مراسم | `ceremonies.view/create/edit/delete/assignments.manage/tasks.manage/payments.manage` |
| پرسنل | `employees.view/create/edit/delete/tasks.manage` |
| مالی | `payments.view/create/edit/delete` |
| پلن‌ها | `plans.view/create/edit/delete` |
| تقویم | `calendar.view/create/edit` |
| گزارشات | `reports.view/export` |
| داشبورد | `dashboard.view/stats` |
| تنظیمات | `settings.view/edit` · `role.manage_system` |

### گارد دقیق (Fine-Grained) بر هر عملیات

هر کامپوننت مدیریتی (`CeremoniesManagement`، `EmployeesManagement`، `PlansManagement`) مجوزهای زیر را جداگانه بررسی می‌کند:

- **ایجاد**: دکمه «جدید» فقط برای دارندگان `*.create` نمایش داده می‌شود
- **ویرایش**: دکمه ویرایش فقط برای دارندگان `*.edit` نمایش داده می‌شود
- **حذف**: دکمه حذف فقط برای دارندگان `*.delete` نمایش داده می‌شود
- اگر کاربر هیچ‌کدام از مجوزهای edit/delete را نداشته باشد، آیکون قفل نمایش داده می‌شود
- حتی اگر دکمه‌ای نمایش داده نشود، در لایه توابع (قبل از فراخوانی API) نیز بررسی مجدد انجام می‌شود

### قوانین تخصیص نقش در مراسم

- **نقش‌های سیستمی** (`admin`, `accountant`, `employee`) هرگز در دراپ‌داون تخصیص نقش مراسم نمایش داده نمی‌شوند.
- فقط نقش‌های **سفارشی** (غیرسیستمی) مثل عکاس، فیلمبردار، مدیر و... می‌توانند به اعضای تیم مراسم اختصاص داده شوند.
- در مدیریت کارمندان: ادمین می‌تواند هر نقشی (حتی سیستمی) را به کارمند اختصاص دهد، اما کاربران عادی فقط نقش‌های غیرسیستمی را می‌بینند.

### داشبورد کاربر معمولی

- کاربران عادی پس از ورود به `/profile` هدایت می‌شوند.
- در پروفایل دو تب مستقل وظایف وجود دارد: **وظایف مراسم** و **وظایف من**
- دکمه هوم در هدر: مدیران → `/admin`، کاربران عادی → `/profile`
- اگر کاربر مجوز یک بخش را نداشته باشد، منوی آن بخش اصلاً نمایش داده نمی‌شود

### سناریوی دسترسی ترکیبی (dav)

- **dav** نقش سراسری: فقط `plans.*`
- **dav** در مراسم X نقش `manager` گرفته → مجوزهای مدیر مراسم (مثل `ceremonies.view`, `ceremonies.tasks.manage`) به JWT اضافه می‌شود
- dav نمی‌تواند به `/admin` برود (middleware بلاک می‌کند)
- dav در `/profile` می‌تواند مراسم X را ببیند و وظایف آن را مدیریت کند
- **مهم**: مجوزهای مراسم در زمان **لاگین** به JWT اضافه می‌شوند. اگر دسترسی پس از لاگین تغییر کند، باید مجدداً وارد شد.

### مجوزهای تقویم

- `calendar.view`: دسترسی به تب تقویم (backward compatible با `ceremonies.view`)
- `calendar.create`: نمایش دکمه رزرو سریع و افزودن مراسم از تقویم
- `calendar.edit`: (برای آینده) ویرایش رویداد مستقیم از تقویم

- `/admin` — فقط `admin`، `accountant` یا کاربران با `isSystem: true`
- `/employee` — هر کاربر احراز هویت‌شده
- `/profile` — هر کاربر احراز هویت‌شده (داشبورد شخصی)
- `/settings` — هر کاربر احراز هویت‌شده (نمایش محتوا بر اساس مجوز)

### مدیریت دسترسی در کد

```ts
// سرور-ساید (API routes)
import { hasGlobalPermission, canManageSystemRole } from '@/lib/accessControl';

// کلاینت‌ساید (components)
import { usePermission } from '@/lib/usePermission';
const { check, AccessDenied } = usePermission();

// در کامپوننت:
const handleAction = () => {
  if (!check(['ceremonies.create'])) return; // نمایش مودال خودکار
  // ادامه عملیات
};

// یا با پرچم‌های دقیق در کامپوننت‌ها:
const canCreate = hasAnyPermission(perms, ['ceremonies.create']);
const canEdit   = hasAnyPermission(perms, ['ceremonies.edit']);
const canDelete = hasAnyPermission(perms, ['ceremonies.delete']);
```

### اضافه کردن مجوزهای جدید

1. کلید جدید را به `DEFAULT_PERMISSIONS` در `lib/permissions.ts` اضافه کنید
2. در پنل تنظیمات → «افزودن مجوزهای پیش‌فرض» را بزنید تا مجوزها در DB ثبت شوند
3. از طریق ماتریس مجوزها، مجوزها را به نقش‌ها اختصاص دهید

## � راهنمای فایل‌های اجرایی (BAT)

این پروژه سه فایل BAT دارد که همه کارها را با دابل‌کلیک انجام می‌دهند:

---

### 🟢 `start.bat` — سرور توسعه (Development)

**کاربرد:** اجرای سریع برای توسعه و تست  
**دابل‌کلیک کن → همه چیز خودکار انجام می‌شود**

**کارهایی که انجام می‌دهد:**
1. اگر سرور قبلی روی پورت 3000 مانده → خودکار می‌کُشد
2. وابستگی‌ها را نصب می‌کند (فقط بار اول)
3. Prisma Client را generate می‌کند
4. دیتابیس SQLite را می‌سازد (اگر نباشد)
5. داده‌های اولیه (seed) را بارگذاری می‌کند
6. سرور توسعه را اجرا می‌کند

**خروجی:** سایت روی `http://localhost:3000`  
**برای بستن:** پنجره CMD را ببند یا `Ctrl+C` بزن

---

### 🔵 `Start Production.bat` — سرور پروداکشن (Production)

**کاربرد:** اجرای سایت در حالت نهایی (سریع‌تر، بهینه‌شده)  
**نیاز:** اول باید `start.bat` حداقل یک‌بار اجرا شده باشد (برای ساخت دیتابیس)

**کارهایی که انجام می‌دهد:**
1. اگر سرور قبلی روی پورت 3000 مانده → خودکار می‌کُشد
2. اگر بیلد وجود نداشت → خودکار `npm run build` می‌زند
3. فایل‌های استاتیک و public را کپی می‌کند
4. دیتابیس را به پوشه standalone کپی می‌کند (با بررسی)
5. IP شبکه را تشخیص می‌دهد
6. سرور پروداکشن را اجرا می‌کند

**خروجی:** سایت روی `http://localhost:3000` + آدرس شبکه  
**برای بستن:** پنجره CMD را ببند یا `Ctrl+C` بزن → سرور بلافاصله متوقف می‌شود

> **نکته مهم:** اگر تغییری در کد دادید، پوشه `.next` را پاک کنید تا بیلد جدید بگیرد.

---

### 🟣 `Build Deploy.bat` — ساخت پکیج Deploy

**کاربرد:** ساخت یک پوشه مستقل برای نصب روی سیستم دیگر  

**کارهایی که انجام می‌دهد:**
1. بیلد پروداکشن می‌گیرد
2. standalone + static + public + دیتابیس را در پوشه `deploy/` کپی می‌کند
3. ماژول `next` را داخل deploy کپی می‌کند (برای سرورهایی که Next.js نصب ندارند)
4. فایل `Start.bat` و `Kill Server.bat` داخل deploy می‌سازد

**محتوای پوشه `deploy/`:**

```
deploy/
├── .next/           (فایل‌های استاتیک و باندل‌های سرور)
├── node_modules/    (شامل ماژول next — برای سرورهای بدون Next.js)
├── prisma/
│   └── dev.db       (دیتابیس SQLite)
├── public/
├── logs/
├── .env
├── Kill Server.bat  (متوقف‌کردن سرور)
├── package.json
├── server.js
└── Start.bat        (راه‌اندازی سرور)
```

**نحوه استفاده:**
1. دابل‌کلیک `Build Deploy.bat`
2. پوشه `deploy/` ساخته می‌شود
3. این پوشه را به هر ویندوزی که **Node.js** دارد کپی کن
4. دابل‌کلیک `Start.bat` داخل پوشه deploy → سایت روی پورت 3000

> **نکته مهم درباره دیتابیس:** `Start.bat` از متغیر `%~dp0` برای تعیین مسیر دقیق دیتابیس استفاده می‌کند تا مستقل از محل پوشه deploy روی هر سیستمی کار کند.

### 🔴 `Kill Server.bat` — توقف سرور

فایل `Kill Server.bat` در پوشه deploy قرار دارد و سرور روی پورت 3000 را متوقف می‌کند.

> **جایگزین:** `node scripts/build-deploy.js` همین کار را انجام می‌دهد.

## 🔧 تغییرات و رفع باگ — نسخه ۱۴۰۵/۰۲/۰۱

### بهبود Deploy و راه‌اندازی روی سرور جدید
- **`Build Deploy.bat`**: ماژول `node_modules/next` به پکیج deploy اضافه شد — دیگر نیازی به نصب Next.js روی سرور مقصد نیست
- **`Build Deploy.bat`**: فایل `Kill Server.bat` به‌صورت خودکار داخل پوشه deploy ساخته می‌شود
- **`deploy/Start.bat`**: متغیر `DATABASE_URL` از مسیر نسبی (`file:./prisma/dev.db`) به مسیر مطلق با `%~dp0` تغییر یافت — رفع خطای `Error code 14: Unable to open the database file` در Next.js standalone mode

### ریشه مشکل و راه‌حل
در Next.js standalone mode، Prisma در worker thread اجرا می‌شود و CWD ممکن است متفاوت باشد؛ به همین دلیل مسیر نسبی `file:./prisma/dev.db` کار نمی‌کند. استفاده از `%~dp0` در bat file مسیر مطلق را به‌درستی تنظیم می‌کند.

## 🔧 تغییرات و رفع باگ — نسخه ۱۴۰۵/۰۱/۳۰

### رفع باگ فایل‌های اجرایی (bat)
- **`Start Production.bat`**: مسیر کپی دیتابیس از `prisma\dev.db` (اشتباه) به `prisma\prisma\dev.db` (صحیح) اصلاح شد
- **`Start Production.bat`**: IP شبکه هاردکد شده (`192.168.20.232`) به تشخیص خودکار تغییر یافت
- **`Start Production.bat`**: خروجی سرور هم‌زمان در کنسول و فایل لاگ نمایش داده می‌شود (قبلاً فقط در فایل لاگ بود)
- **`Build Deploy.bat`**: متغیر `DATABASE_URL` به `Start.bat` تولیدشده اضافه شد (بدون آن سرور خطا می‌داد)
- **`Build Deploy.bat`**: مسیر کپی دیتابیس به `deploy\prisma\dev.db` اصلاح شد
- **`scripts/build-deploy.js`**: انکودینگ `Start.bat` از `ascii` به `utf-8` تغییر یافت (کاراکترهای فارسی خراب نشوند)
- **`scripts/build-deploy.js`**: `DATABASE_URL` و `COOKIE_SECURE` به اسکریپت استارت اضافه شد

### رفع مشکلات امنیتی
- **سطح دسترسی API**: بررسی نقش ادمین به تمام عملیات نوشتنی (ایجاد/ویرایش/حذف) اضافه شد:
  - مراسمات: `POST`، `PUT`، `DELETE`
  - پرداخت‌ها: `POST`
  - تخصیص کارمندان: `POST`، `DELETE`
  - پلن‌ها: `POST`، `PUT`، `DELETE`
  - تنظیمات: `PUT`
  - ویرایش کارمندان: `PUT`
- **`isAdmin`**: نقش `accountant` (حسابدار) نیز دسترسی مدیریتی دارد (هماهنگ با middleware)
- **`register` API**: رمز عبور هش‌شده دیگر در پاسخ ارسال نمی‌شود
- **`logout` API**: فلگ `secure` کوکی با `COOKIE_SECURE` محیطی هماهنگ شد (قبلاً `NODE_ENV` چک می‌شد)
- **حذف کارمند**: عملیات در یک `$transaction` انجام می‌شود (جلوگیری از داده‌های یتیم)

### رفع باگ‌های عملکردی
- **مراسمات کارمند**: رفع باگ تکرار مراسمات هنگامی که کارمند چند وظیفه در یک مراسم دارد
- **ورود ارقام فارسی**: در فرم مراسمات، ارقام فارسی (۰-۹) و عربی (٠-٩) به‌درستی تبدیل به عدد لاتین می‌شوند
- **ساعت حضور**: رفع `parseFloat("14:30") = 14` — فرمت `HH:MM` به عدد اعشاری تبدیل می‌شود (مثلاً `14:30` → `14.5`)
- **SWC Build**: پکیج `@next/swc-win32-x64-msvc` نسخه ناسازگار (16.x) حذف شد — بیلد به‌درستی انجام می‌شود

### دیباگ روی گوشی واقعی

از موبایل از طریق IP شبکه وصل شوید (هنگام اجرا نمایش داده می‌شود):

```
http://192.168.x.x:3000
```

لاگ ورودها در `logs/access.log`:

```
[2026-04-19T10:23:11Z] LOGIN OK   user="admin" role=admin IP:192.168.20.15
[2026-04-19T10:23:15Z] LOGIN FAIL user="admin" IP:192.168.20.15
```

اگر از موبایل لاگین نمی‌شود:
1. `logs/access.log` را بررسی کنید
2. `LOGIN FAIL` → رمز اشتباه
3. هیچ entry → موبایل به سرور دسترسی ندارد (IP چک شود)
4. `LOGIN OK` ولی ریدایرکت نمی‌شود → مطمئن شوید `COOKIE_SECURE=false` در `.env` است
